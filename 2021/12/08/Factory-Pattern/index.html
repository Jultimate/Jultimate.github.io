Factory Pattern

发表于 2021-12-08 更新于 2021-12-18 分类于 Java
设计模式之工厂模式：1、简单工厂模式 2、抽象工厂模式

简单工厂模式

Play接口

public interface Play
{
    void play();
}

接口实现类

public class VideoGames implements Play
{
    @Override
    public void play()
    {
        System.out.println("Play computer and video games.");
    }
}

public class PingPong implements Play
{
    @Override
    public void play()
    {
        System.out.println("Play ping-pong.");
    }
}

public class Playground implements Play
{
    @Override
    public void play()
    {
        System.out.println("children shouting and running in the playground.");
    }
}

Play工厂

public class PlayFactory
{
    public static Play getPlay(String type)
    {
        switch(type)
        {
            case "VIDEO_GAMES":
                return new VideoGames();
            case "PING_PONG":
                return new PingPong();
            case "PLAYGROUND":
                return new Playground();
        }
    }
}

简单工厂模式属于建造者模式，工厂类已经帮我们new好了对象，简单工厂模式有很大的弊端，如果有新的Play接口实现类的话，我们必须更改工厂类的代码，这违背了设计原则中的开闭原则（对修改关闭，对扩展开放）；而且后续实现类越来越多会造成工厂类异常臃肿，职责过于集中。

工厂模式是根据现实中的工厂所抽象出来的，可以看到，目前我们的工厂只有一条“流水线”，无法生产不同“类型”的产品，这时候就需要引入抽象工厂模式。

抽象工厂模式

接口

public interface Play
{
    void play();
}

public interface Read
{
    void read();
}

各种Play

public class Soccer implements Play
{
    @Override
    public void play()
    {
        System.out.println("I played soccer when I was younger.");
    }
}

public class GrandTheftAuto implements Play
{
    @Override
    public void play()
    {
        System.out.println("You forget a thousand things every day，How about you make sure this is one of them?");
    }
}

public class Battlefield implements Play
{
    @Override
    public void play()
    {
        System.out.println("They push,we push.If every once in a while,we push hard enough that the world beyond the war glimmers,just out of reach. ");
    }
}

各种Read

public class OperatingSystems implements Read
{
    @Override
    public void read()
    {
        System.out.println("Thus the real point of the educational process: to go forth, to study many new and fascinating topics, to learn, to mature, and most importantly, to find something that lights a fire for you.");
    }
}

public class A_GlobalHistory implements Read
{
    @Override
    public void read()
    {
        System.out.println("A Global History: From Prehistory to the 21st Century");
    }
}

抽象工厂

public abstract class AbstractFactory
{
    protected Play getPlay(String type);

    protected Read getRead(String type); 
}

各种车间

public class PlayFactory extends AbstractFactory
{
    @Override
    protected Play getPlay(String type)
    {
        switch(type)
        {
            case "SOCCER":
                return new Soccer();
            case "GRAND_THEFT_AUTO":
                return new GrandTheftAuto();
            case "BATTLEFIELD":
                return new Battlefield();
        }
    }
}

public class ReadFactory extends AbstractFactory
{
    @Override
    protected Read getRead(String type)
    {
        switch(type)
        {
            case "OS":
                return new OperatingSystems();
            case "HISTORY":
                return new A_GlobalHistory();
        }
    }
}

现在我们的工厂可以生产两种不同的产品了，在这个程序中，我们的工厂拥有两个不同的“车间”，可以生产出不同种类的产品：Play，Read。（虽然这种情况不太可能发生在现实生活中）

抽象工厂模式可以看作是简单工厂模式的一个整合，通过将这些简单工厂整合起来，形成一个更大，功能更为全面的工厂，即我们定义的抽象工厂，在这种模式下，被我们通过抽象类整合进来的简单工厂可以看作一条“流水线”或一个“车间”，由它们共同组成了可以生产多种类型产品的抽象工厂。

同时，一些问题在抽象工厂中仍无法得到解决：

如果扩充新类型的产品，需要对整个抽象工厂模式的代码做出调整
某类型的产品如果需要增加新的具体实现，需要对单个工厂（或车间）的代码做出调整
